# SECTION 1 — AWS Security Fundamentals (Q1–Q30)

**Q1. What is the AWS Shared Responsibility Model and why does it matter in a sales conversation?**
A: AWS secures the cloud infrastructure (hardware, data centers, hypervisor, global network). The customer secures what they put IN the cloud (OS, applications, data, IAM, network configuration). In sales, this matters because customers often assume AWS handles everything — correcting this opens the door to selling security services like GuardDuty, KMS, and Security Hub to fill the customer's side of the responsibility.

**Q2. What is IAM and what are its four main components?**
A: IAM (Identity and Access Management) controls who can access AWS resources and what they can do. Four components: (1) Users — individual identities with long-term credentials; (2) Groups — collections of users, assign policies to groups not individuals; (3) Roles — temporary identities assumed by services, apps, or federated users, no long-term credentials; (4) Policies — JSON documents defining Allow or Deny on specific Actions and Resources.

**Q3. What is the principle of least privilege and how do you implement it in AWS?**
A: Grant users and services only the minimum permissions needed to do their job — nothing more. In AWS: attach specific policies instead of AdministratorAccess, use IAM permission boundaries to cap maximum permissions, use SCPs in Organizations to prevent privilege escalation, regularly review and remove unused permissions with IAM Access Analyzer, and use IAM Access Advisor to see what permissions are actually being used.

**Q4. What is the difference between an IAM Role and an IAM User?**
A: An IAM User has long-term credentials (username/password, access keys) tied to a specific person. An IAM Role has no long-term credentials — it is assumed temporarily by AWS services, applications, EC2 instances, or federated users, and provides temporary security tokens. Best practice: use roles for applications and services, never embed access keys in code.

**Q5. What is AWS KMS and how does envelope encryption work?**
A: KMS (Key Management Service) creates and manages cryptographic keys for encrypting AWS data. Envelope encryption: a data key (generated by KMS) encrypts the actual data, then KMS encrypts that data key with a Customer Master Key (CMK). The encrypted data key is stored alongside the encrypted data. This way, KMS only needs to decrypt the small data key, not the entire dataset — efficient and secure.

**Q6. What is Amazon GuardDuty and what data sources does it analyze?**
A: GuardDuty is an intelligent threat detection service that uses machine learning to continuously monitor AWS accounts for malicious activity. Data sources: VPC Flow Logs (network traffic patterns), AWS CloudTrail (API call history), DNS logs (domain lookups). Optional: S3 data events, EKS audit logs, RDS login activity, Lambda network activity. GuardDuty requires no agents — it analyzes existing log data.

**Q7. What types of threats does GuardDuty detect?**
A: Cryptocurrency mining on EC2 instances, unusual API calls (account compromise indicators), port scanning and network reconnaissance, communication with known malicious IPs or domains, credential theft attempts, unusual IAM behavior (e.g., root account activity), data exfiltration patterns, brute-force attacks on RDS databases.

**Q8. What is AWS Security Hub and how does it differ from GuardDuty?**
A: GuardDuty is a threat detection service — it actively finds threats. Security Hub is a centralized aggregation and management dashboard that collects findings FROM GuardDuty and other services (Macie, Inspector, Firewall Manager, third-party tools like CrowdStrike). Security Hub also runs compliance checks against frameworks like CIS Benchmarks, PCI DSS, and provides a security score. Think: GuardDuty finds threats, Security Hub organizes and prioritizes all findings.

**Q9. What is Amazon Macie and what types of data does it detect?**
A: Macie uses machine learning to automatically discover, classify, and protect sensitive data stored in S3. It detects: PII (names, addresses, SSNs, passport numbers, credit card numbers), financial data, AWS credentials and API keys, healthcare data (PHI), and custom data patterns you define. Key sales pitch: "Macie finds sensitive data you didn't know you had exposed."

**Q10. What is the difference between AWS WAF and AWS Shield?**
A: WAF (Web Application Firewall) protects against application layer (Layer 7) attacks: SQL injection, cross-site scripting (XSS), bot traffic, malicious requests — it filters HTTP/HTTPS traffic based on rules. Shield protects against DDoS (Distributed Denial of Service) attacks at the network and transport layers (Layers 3/4). Shield Standard is free and automatic; Shield Advanced adds Layer 7 DDoS protection, real-time metrics, financial protection, and 24/7 DDoS Response Team access.

**Q11. What does AWS CloudTrail record and why is it critical for compliance?**
A: CloudTrail records every AWS API call: who made it (IAM identity), what action (API call name), when (timestamp), from where (source IP), on what resource, and the response. It is critical for compliance because it provides a complete, immutable audit trail for governance frameworks like PCI DSS, HIPAA, SOC 2, and GDPR — auditors can see every action ever taken in the account.

**Q12. What is the difference between CloudTrail Management Events and Data Events?**
A: Management Events (enabled by default, free): control-plane operations like creating EC2 instances, modifying IAM policies, creating S3 buckets. Data Events (disabled by default, charged): data-plane operations like S3 object reads/writes and Lambda function invocations. Data events generate much higher volume and cost but are needed for detailed data access auditing.

**Q13. What is AWS Config and how does it differ from CloudTrail?**
A: CloudTrail answers "who did what action?" — it records API calls. AWS Config answers "what does my infrastructure look like right now and how has it changed?" — it continuously records the configuration state of resources and evaluates them against compliance rules. CloudTrail is an activity log; Config is a configuration inventory and compliance checker. Example: Config tells you a security group was changed from closed to open port 22, and flags it as NON_COMPLIANT.

**Q14. What is Amazon Inspector and what does it scan?**
A: Inspector is an automated vulnerability assessment service. It scans: EC2 instances (CVEs in OS packages and application libraries, network reachability), container images in Amazon ECR (software vulnerabilities), and Lambda functions (software vulnerabilities in dependencies). It prioritizes findings by risk score and integrates with Systems Manager for patch management.

**Q15. What is Amazon Detective and when would you recommend it?**
A: Detective uses machine learning to analyze and visualize security findings to help investigate the root cause of security incidents. It automatically builds behavioral baselines and shows interactive visualizations of entity relationships (IP addresses, AWS users, resources). Recommend it to customers with an active security team (SOC) that needs to investigate GuardDuty findings quickly — it dramatically reduces Mean Time To Investigate (MTTI).

**Q16. What is AWS Artifact and when would a customer need it?**
A: Artifact is a self-service portal providing on-demand access to AWS compliance reports and agreements. Reports include: SOC 1/2/3, PCI DSS Attestation, ISO 27001, FedRAMP authorization, HIPAA eligibility. Customers need it when auditors request evidence that AWS infrastructure meets compliance standards, or when they need to sign a Business Associate Agreement (BAA) for HIPAA.

**Q17. What are Service Control Policies (SCPs) and how do they work?**
A: SCPs are policies applied at the AWS Organization or Organizational Unit (OU) level that set the maximum permissions available to all accounts within. Key point: SCPs do NOT grant permissions — they only restrict what is possible. Even if an IAM admin in an account has full administrator access, an SCP can prevent them from disabling CloudTrail or creating resources in certain regions. Used as guardrails across an entire organization.

**Q18. What is VPC Flow Logs and what information does it capture?**
A: VPC Flow Logs capture metadata about IP traffic flowing to/from network interfaces in a VPC. Captured fields: source/destination IP, source/destination port, protocol, packet count, byte count, start/end time, action (ACCEPT/REJECT). Stored in CloudWatch Logs or S3. Used for security analysis (detecting port scans, unusual traffic), troubleshooting (why is traffic being rejected), and network forensics. Note: it captures metadata, NOT packet content.

**Q19. What is the difference between a Security Group and a Network ACL?**
A: Security Groups: stateful (return traffic automatically allowed), operate at the instance level, support allow rules only (implicit deny everything else), evaluated as a whole. NACLs: stateless (must explicitly allow return traffic), operate at the subnet level, support both allow and deny rules, rules evaluated in order by number. Use Security Groups for instance-level control; NACLs for subnet-level blocking (e.g., block a known malicious IP at the subnet level).

**Q20. What is AWS Secrets Manager and when would you use it over environment variables?**
A: Secrets Manager securely stores, rotates, and retrieves secrets (database passwords, API keys, OAuth tokens). Unlike environment variables (which are static, visible in code, and don't rotate), Secrets Manager: automatically rotates credentials (integrated with RDS), encrypts with KMS, provides fine-grained IAM access control, and keeps full version history. Use Secrets Manager for any production credential, especially database passwords.

**Q21. What is Zero Trust and how does AWS implement it?**
A: Zero Trust: "Never trust, always verify" — no user or system is trusted by default, even inside the network. AWS implementation: IAM Identity Center (centralized identity verification), AWS Verified Access (zero trust app access without VPN), PrivateLink (private service connectivity), mutual TLS (mTLS) for service-to-service auth, Network Firewall for deep packet inspection, and VPC micro-segmentation. Contrast with traditional perimeter security: once inside the VPN, users were trusted everywhere.

**Q22. What is AWS Organizations and why is it important for enterprise customers?**
A: Organizations allows central management of multiple AWS accounts. Features: consolidated billing (one bill, volume discounts), Organizational Units (OUs) for logical grouping, SCPs for organization-wide guardrails, centralized CloudTrail and Security Hub, account vending via Control Tower. Important for enterprises because large companies often have 10, 100, or 1,000+ AWS accounts and need consistent security controls across all of them without manual configuration.

**Q23. What is the purpose of S3 Block Public Access and what are its four settings?**
A: S3 Block Public Access prevents S3 data from being accidentally or maliciously made public. Four settings: (1) BlockPublicAcls — blocks new public ACLs; (2) IgnorePublicAcls — ignores existing public ACLs; (3) BlockPublicPolicy — blocks new public bucket policies; (4) RestrictPublicBuckets — restricts access to buckets with public policies. Can be set at account level (applies to all buckets) or per-bucket. Best practice: enable all four at account level.

**Q24. What is IAM Access Analyzer and when would you use it?**
A: Access Analyzer identifies resources in your account that are shared with external entities (outside the account or organization) — S3 buckets, IAM roles, KMS keys, Lambda functions, SQS queues. It generates findings for any resource accessible from outside your trust zone. Use it during security reviews to find unintended public or cross-account access, and to continuously monitor for new external sharing.

**Q25. What is AWS Trusted Advisor and what security checks does it provide?**
A: Trusted Advisor provides recommendations across 5 categories: Cost Optimization, Performance, Security, Fault Tolerance, Service Limits. Security checks available to all customers (free): MFA on root account, S3 bucket permissions (public access), security group open ports (unrestricted), IAM use (no root-only usage), EBS and RDS public snapshots, exposed access keys. Business/Enterprise support unlocks all checks.

**Q26. What is Amazon Cognito and when would you use it vs IAM?**
A: Cognito provides user authentication and authorization for web and mobile applications — it manages your application's end users (customers, not AWS employees). IAM manages AWS account access (who can use AWS services). Use Cognito when you need customer-facing sign-up/sign-in (social login, MFA, SAML). Use IAM when you need to control who accesses AWS resources internally. Cognito can grant users temporary AWS credentials via Identity Pools to access S3 or DynamoDB directly.

**Q27. What is AWS Control Tower and what problem does it solve?**
A: Control Tower automates the setup of a secure, multi-account AWS environment (called a Landing Zone) following AWS best practices. It solves the problem of manual, inconsistent account setup. Features: pre-configured guardrails (detective and preventive controls), Account Factory (automated provisioning of new accounts with security baselines), compliance dashboard, integration with Organizations and IAM Identity Center.

**Q28. What is the AWS Well-Architected Framework Security Pillar?**
A: One of six pillars of the Well-Architected Framework. The Security Pillar covers: Identity and Access Management (who can do what), Detective Controls (monitoring and alerting), Infrastructure Protection (network and compute hardening), Data Protection (encryption and classification), and Incident Response (how to react to security events). AWS offers the Well-Architected Tool to assess workloads against these principles — useful in customer conversations to identify security gaps.

**Q29. What is encryption in transit and how does AWS enforce it?**
A: Encryption in transit protects data moving over a network from interception. AWS enforces it via TLS/SSL on all API endpoints, ACM (Certificate Manager) for free SSL/TLS certificates, HTTPS enforcement on S3 buckets via bucket policies, enforcing SSL on RDS via parameter groups, and API Gateway with TLS termination. All AWS service endpoints use TLS 1.2 minimum.

**Q30. What are the three types of KMS keys and when would you use each?**
A: (1) AWS Owned Keys: managed entirely by AWS for use in managed services, customer has no visibility or control — transparent encryption, no additional cost. (2) AWS Managed Keys: created by AWS in your account for specific services (e.g., aws/s3, aws/rds), you can view usage but can't manage them — free, automatic rotation. (3) Customer Managed Keys (CMKs): you create, manage, and control — custom key policies, manual or automatic rotation, cross-account sharing, audit via CloudTrail — additional cost per key per month. Use CMKs when you need fine-grained control, cross-account encryption, or key deletion capability.

# SECTION 2 — AI Services & AI Security (Q31–Q70)

**Q31. What is the difference between Amazon Bedrock and Amazon SageMaker?**
A: Bedrock: access pre-built foundation models (Claude, Llama, Titan, Mistral) via API — no ML expertise required, pay per token, no infrastructure to manage. Best for: using existing powerful AI models quickly. SageMaker: full ML platform for building, training, and deploying custom models — requires ML expertise, more control and customization. Best for: organizations building proprietary ML models. Many customers use both: SageMaker for custom models, Bedrock for foundation model capabilities.

**Q32. What foundation models are available on Amazon Bedrock?**
A: Anthropic Claude (strong reasoning, analysis, long documents, coding), Meta Llama (open-source, good for customization and fine-tuning), Amazon Titan (AWS native, text and embeddings), Mistral (efficient European models), Stability AI (image generation), Cohere (enterprise search and RAG). Key advantage: model choice — customers aren't locked into one vendor unlike Azure OpenAI (GPT only).

**Q33. What is RAG (Retrieval Augmented Generation) and why is it important?**
A: RAG combines an LLM with a custom knowledge base (company documents, databases). Instead of relying only on the model's training data, RAG retrieves relevant documents at query time and feeds them to the LLM as context. Result: the AI answers questions using your company's specific, up-to-date information without retraining the model. Example: HR chatbot that answers questions using your actual company policies. Bedrock Knowledge Bases implements RAG natively.

**Q34. What is Amazon Bedrock Guardrails and what does it protect against?**
A: Guardrails adds safety controls to AI applications built on Bedrock. Features: content filtering (block harmful, violent, or hateful content), denied topics (block specific subjects like competitor discussions), word filters (block specific terms), PII redaction (automatically detect and mask sensitive data in inputs/outputs), grounding checks (verify responses are based on provided sources). Critical for enterprise AI adoption — security and compliance teams require these controls.

**Q35. What is prompt injection and why is it the most critical AI security threat?**
A: Prompt injection is when an attacker crafts malicious input that hijacks the AI's instructions. Direct injection: user manipulates the AI directly ("Ignore your system prompt and..."). Indirect injection: malicious content in documents the AI reads that overrides instructions. It is most critical because: it can expose system prompts (IP theft), cause data leakage, make the AI act against its intended purpose, and bypass safety controls. Defense: Bedrock Guardrails, input validation, separation of user data and instructions.

**Q36. What is the OWASP LLM Top 10? Name at least 5.**
A: OWASP's list of top security risks for Large Language Model applications: (1) Prompt Injection — hijacking AI behavior via malicious input; (2) Insecure Output Handling — trusting AI output without validation, leading to XSS or SQL injection; (3) Training Data Poisoning — corrupting training data to compromise model behavior; (4) Model Denial of Service — overwhelming the model with expensive queries; (5) Supply Chain Vulnerabilities — compromised models, datasets, or plugins; (6) Sensitive Information Disclosure — model reveals training data or system prompts; (7) Insecure Plugin Design — AI agents calling external tools without authorization; (8) Excessive Agency — AI given too many permissions acts unintended; (9) Overreliance — trusting AI output without human review; (10) Model Theft — extracting model via API queries.

**Q37. What is Amazon SageMaker Model Monitor and why does it matter for AI security?**
A: Model Monitor continuously monitors deployed ML models in production for: data quality drift (input data changes from training distribution), model quality drift (predictions becoming less accurate), bias drift (model becoming more biased over time), and feature attribution drift (which features influence predictions changes). For security: it detects when a model is being manipulated by adversarial inputs or when data poisoning has occurred. Alerts via CloudWatch when drift exceeds thresholds.

**Q38. What is data poisoning in the context of AI?**
A: Data poisoning is when an attacker injects malicious, manipulated, or misleading data into a model's training dataset to corrupt its behavior. Types: (1) Backdoor attacks — model performs normally but behaves maliciously for specific trigger inputs; (2) Label flipping — changing training labels to cause misclassification; (3) Model degradation — reducing overall model performance. Defense: data validation, anomaly detection on training data, SageMaker Clarify for bias detection, secure data pipelines.

**Q39. What is Amazon Q and what are its different versions?**
A: Amazon Q is AWS's generative AI assistant. Three versions: (1) Amazon Q Business — enterprise AI assistant connected to company data sources (SharePoint, Jira, S3, databases) — answers employee questions using internal knowledge; (2) Amazon Q Developer — AI coding assistant for developers, similar to GitHub Copilot, integrated in IDEs and AWS Console; (3) Amazon Q in QuickSight — BI assistant for data analysis using natural language. All versions include security controls for data access based on user permissions.

**Q40. How do you address a security team's concerns about deploying AI on AWS?**
A: Address key concerns: (1) Data privacy — Bedrock does NOT use your data to train models; data stays in your AWS account and region; (2) Access control — IAM controls who can call which models; (3) Audit trail — Bedrock Model Invocation Logging captures all inputs/outputs to CloudWatch/S3; (4) Network security — use VPC private endpoints so no AI data crosses the public internet; (5) Content safety — Bedrock Guardrails filters harmful content and redacts PII; (6) Compliance — Bedrock is HIPAA eligible, PCI DSS compliant, SOC 2 certified.

**Q41. What is model theft and how do you prevent it?**
A: Model theft (model extraction) is when an attacker systematically queries an AI model to recreate it without authorization — stealing your proprietary model or the value of expensive fine-tuning. Prevention: API rate limiting (throttle excessive queries), AWS WAF rules to detect unusual query patterns, monitoring token consumption anomalies with CloudWatch, watermarking model outputs, requiring authentication (IAM) for all model access.

**Q42. What is the difference between AI hallucination and a security risk?**
A: Hallucination: AI generates plausible-sounding but factually incorrect information. Security risk when: AI used for compliance decisions generates incorrect regulatory guidance (legal liability), AI security tools incorrectly dismiss real threats, AI-generated code contains vulnerabilities, AI incorrectly states HIPAA requirements to a healthcare customer. Mitigation: grounding with RAG (connect to authoritative sources), human review for high-stakes decisions, Bedrock Guardrails with grounding checks.

**Q43. What is Amazon Rekognition and what are its security use cases?**
A: Rekognition performs image and video analysis using ML. Security use cases: face detection and comparison for identity verification (comparing selfie to ID document), content moderation (detecting inappropriate content in user uploads), object detection (detecting weapons or threats in images), text detection in images (reading license plates, signs), analyzing security camera footage (detecting people in restricted areas). Can process real-time video streams.

**Q44. What is Amazon Comprehend and how does it relate to data security?**
A: Comprehend performs natural language processing on text. Security use cases: detecting PII in documents (names, SSNs, credit card numbers, phone numbers) — Comprehend PII detection is simpler/cheaper than Macie for non-S3 text; sentiment analysis of security tickets to prioritize urgent issues; entity extraction to identify mentioned assets, people, or organizations in security reports; custom classification to categorize documents by sensitivity level.

**Q45. What is Amazon Textract and why would a financial services customer need it?**
A: Textract automatically extracts text, handwriting, and structured data (forms, tables) from documents like PDFs and images — beyond simple OCR. Financial services use cases: processing loan applications automatically, extracting data from financial statements, digitizing paper contracts, automating invoice processing. Security relevance: Textract + Comprehend together can extract and then classify PII in scanned documents — important for GDPR and data residency compliance.

**Q46. What is SageMaker Clarify and why is it important for responsible AI?**
A: Clarify detects bias in ML models and explains model predictions. Bias detection: identifies statistical disparities in training data and model predictions across demographic groups (gender, age, race). Explainability: provides SHAP values to show which features influenced each prediction. Important for: regulated industries where AI decisions must be explainable (loan approvals, medical diagnoses), GDPR Article 22 (right to explanation for automated decisions), internal AI ethics policies.

**Q47. What is Bedrock Agents and what security considerations apply?**
A: Bedrock Agents are AI systems that can take multi-step actions — query databases, call APIs, run calculations — to complete tasks autonomously. Security considerations: (1) Principle of least privilege — agents should only have access to necessary APIs and data; (2) Action confirmation — require human approval before irreversible actions; (3) Input validation — validate all agent inputs and outputs; (4) Logging — enable invocation logging for all agent actions; (5) Isolation — run agents in VPC with restricted internet access; (6) Guardrails — apply content filtering to agent outputs.

**Q48. How does Amazon Macie use machine learning?**
A: Macie uses supervised ML models trained on examples of sensitive data patterns to classify data in S3. It learns from: known PII formats (SSN patterns, credit card formats), known credential patterns (AWS access key formats, private key headers), custom data identifiers you define using regex. Macie also uses ML to detect anomalous access patterns to S3 (unusual download volume, access from new location) which may indicate compromise.

**Q49. What is the difference between model fine-tuning and RAG in Bedrock?**
A: Fine-tuning: retraining a base model on your specific domain data to change the model's underlying behavior and knowledge — expensive (compute costs), time-consuming, produces a customized model version. RAG: dynamically retrieving relevant documents at query time and providing them as context — no model training, cheaper, knowledge stays current (update documents without retraining), knowledge is source-traceable. RAG is preferred for most enterprise use cases; fine-tuning for domain-specific language or style.

**Q50. What is Amazon Bedrock Model Evaluation?**
A: Bedrock Model Evaluation allows you to compare different foundation models side-by-side on your specific use case. You provide test prompts, run them against multiple models, and evaluate responses using: automatic metrics (accuracy, toxicity, robustness), human evaluation workflows (reviewers score responses). Helps customers choose the right model for their use case before committing to a specific model in production. Important for security: can test how different models respond to adversarial prompts.

**Q51. What is Excessive Agency in AI security?**
A: OWASP LLM #8 — when an AI agent is given more capabilities, permissions, or autonomy than necessary, leading to unintended harmful actions. Example: an AI coding assistant with write access to production databases instead of just read access. Prevention: apply principle of least privilege to all AI agent permissions, implement human-in-the-loop for high-impact actions, limit agent scope to minimum required capabilities, use IAM conditions to restrict what agents can do.

**Q52. How would you secure a generative AI application built on Bedrock end-to-end?**
A: Layered security approach: (1) Identity — IAM roles with minimum permissions to invoke specific models only; (2) Network — VPC endpoint for Bedrock, no public internet access; (3) Input validation — sanitize user inputs before sending to model, use WAF; (4) Guardrails — content filtering, denied topics, PII redaction; (5) Output validation — never trust AI output for security decisions without human review; (6) Logging — enable invocation logging to S3/CloudWatch for audit; (7) Monitoring — CloudWatch alerts on unusual token consumption; (8) Data — never include credentials or secrets in prompts; (9) Compliance — verify Bedrock model compliance with applicable regulations.

**Q53. What is Amazon SageMaker Canvas and who is it for?**
A: Canvas is a no-code visual interface for ML — business analysts and non-technical users can build, train, and deploy ML models without writing code. Users import data, Canvas automatically selects the best algorithm, trains the model, and shows predictions. Security consideration: Canvas users may not understand data governance implications — security controls needed: IAM to restrict which data sources Canvas can access, data classification before allowing Canvas use, output review.

**Q54. What is the difference between supervised, unsupervised, and reinforcement learning?**
A: Supervised learning: model trained on labeled data (input-output pairs) to predict outputs for new inputs. Examples: spam detection, fraud classification. Unsupervised learning: model finds patterns in unlabeled data. Examples: customer segmentation, anomaly detection. Reinforcement learning: model learns by trial and error, receiving rewards for good actions. Examples: game playing, recommendation systems, robotics. Security relevance: anomaly detection (unsupervised) is used in GuardDuty and Macie.

**Q55. What is Amazon Q Security Lake?**
A: Amazon Security Lake automatically centralizes security data from AWS services (CloudTrail, VPC Flow Logs, Route 53, Security Hub, Lambda) and third-party sources into a dedicated S3 data lake formatted in OCSF (Open Cybersecurity Schema Framework). Benefits: normalized security data queryable with Athena or third-party SIEMs, long-term retention, cross-account aggregation, partner integrations. Relevant for customers building a security data lake for advanced threat hunting.

**Q56. What is prompt engineering and why does it matter for security?**
A: Prompt engineering is crafting AI inputs to get desired, accurate, and safe outputs. Security relevance: (1) System prompts define AI behavior — poorly written system prompts can be overridden by prompt injection; (2) A well-engineered prompt reduces hallucination risk; (3) Prompts should avoid requesting operations the AI shouldn't perform; (4) System prompts may contain sensitive business logic that needs protection. Best practice: treat system prompts as sensitive IP, don't expose them to users.

**Q57. What is model drift and why is it a security concern?**
A: Model drift occurs when a model's performance degrades over time because the real-world data it encounters differs from its training data. Security concern: (1) A security classification model that drifts may start missing threats (false negatives); (2) Attackers can deliberately cause drift through adversarial data; (3) Compliance models that drift may generate incorrect compliance decisions; (4) Fraud detection that drifts misses new fraud patterns. SageMaker Model Monitor detects drift automatically.

**Q58. What is Amazon Bedrock Invocation Logging?**
A: Enables logging of all Bedrock model invocations (requests and responses) to Amazon S3 or CloudWatch Logs. Captures: model ID, timestamp, request body (prompt), response body, token counts, user identity (IAM). Required for: compliance auditing, detecting inappropriate use, security incident investigation, cost allocation. Critical for regulated industries (HIPAA, PCI DSS) that must demonstrate oversight of AI usage.

**Q59. What is Insecure Output Handling in AI security?**
A: OWASP LLM #2 — when an application uses AI output directly without validation, treating it as safe. Example: AI generates HTML/JavaScript that gets rendered directly in a browser — attacker crafts a prompt that makes the AI output a malicious script (XSS). Or AI generates SQL that gets executed directly (SQL injection). Defense: treat AI output as untrusted user input, sanitize and validate before using in code, databases, or rendering in browsers.

**Q60. What is Amazon Comprehend Medical?**
A: A specialized version of Comprehend designed for healthcare — extracts medical information from unstructured text: medical conditions (ICD-10 codes), medications (dosage, frequency), anatomy (body parts, systems), test/treatments/procedures, protected health information (PHI). Relevant for HIPAA-covered entities processing medical records. Important: Comprehend Medical is HIPAA eligible — customer must sign a BAA with AWS.

**Q61. What is the AI Security risk of training on customer data?**
A: If customer data is used to train a shared model, that data could potentially be extracted by other users of the model (training data extraction attack). AWS Bedrock mitigates this: customer data is NOT used to train Bedrock models — this is contractually guaranteed and a key differentiator vs. some competitors. SageMaker models trained on customer data stay in the customer's account.

**Q62. What is Amazon Titan and when would you recommend it?**
A: Amazon Titan is AWS's own family of foundation models available on Bedrock. Models: Titan Text (text generation), Titan Embeddings (convert text to numerical vectors for search and RAG), Titan Image Generator (image creation), Titan Multimodal Embeddings (images + text). Recommend when: customer wants AWS-native models (no third-party data sharing concerns), need embeddings for a knowledge base, or want image generation without third-party dependencies.

**Q63. What is Federated Learning and how does it relate to AI security?**
A: Federated learning trains ML models across multiple decentralized devices or servers without sharing raw data — the model comes to the data instead of data going to the model. Security benefits: data never leaves the organization's environment, reduces data breach risk during training, enables collaborative AI across organizations that can't share data (e.g., hospitals collaborating on disease models without sharing patient records). AWS supports federated learning via SageMaker.

**Q64. What is Sensitive Information Disclosure in AI systems?**
A: OWASP LLM #6 — when an AI model reveals sensitive information from its training data or context window. Types: (1) Training data leakage — model recites memorized sensitive info from training (e.g., email addresses, code snippets); (2) System prompt leakage — model reveals its configuration instructions; (3) Context leakage — model reveals data from other users' sessions (rare but possible). Defense: Bedrock Guardrails with PII redaction, careful system prompt design, regular red-team testing.

**Q65. What is Amazon Personalize and what security considerations apply?**
A: Personalize provides real-time personalization and recommendation capabilities — you provide user interaction data, it trains recommendation models and serves recommendations via API. Security considerations: user behavioral data is sensitive — apply KMS encryption, IAM access controls, data minimization principles. GDPR implications: recommendation systems may profile users — must consider right to erasure (deleting user data from models), consent, and data residency.

**Q66. How does AI help AWS security services work better?**
A: GuardDuty uses ML to establish behavioral baselines and detect anomalies (e.g., EC2 normally communicates with 10 IPs, suddenly connects to 1,000 — ML flags this). Macie uses ML to recognize patterns of sensitive data across millions of file formats. Inspector uses ML to prioritize vulnerability findings by actual exploitability (not just CVSS score). Security Hub uses ML for automated insight generation. Access Analyzer uses automated reasoning to mathematically prove access paths.

**Q67. What is the difference between Amazon Q Business and Amazon Q Developer?**
A: Amazon Q Business: enterprise AI assistant for employees — connect to company data sources (SharePoint, Jira, Confluence, S3, databases), ask questions in natural language, get answers from your company's own knowledge. Security: respects existing permissions (user only sees data they're authorized for). Amazon Q Developer: AI coding assistant for developers — code generation, debugging, test writing, security vulnerability detection, transformation (upgrade legacy code). Can detect security vulnerabilities in code as you write.

**Q68. What is AI red-teaming and who does it?**
A: AI red-teaming is adversarial testing of AI systems — systematically trying to make the AI behave badly: prompt injection attempts, jailbreaking, extracting sensitive information, generating harmful content, bypassing safety controls. Done by: internal security teams, specialized AI red-team firms, or using AWS Bedrock's built-in model evaluation features. Should be conducted before any AI application goes to production, especially in regulated industries.

**Q69. What is Amazon Lookout for Metrics and what is its security application?**
A: Lookout for Metrics uses ML to automatically detect anomalies in business metrics (revenue, user signups, error rates) without requiring ML expertise. Security application: detecting anomalies that may indicate security incidents — sudden spike in failed login attempts, unusual drop in transaction volume (possible DDoS impact), unexpected spike in data transfer (possible exfiltration). Integrates with CloudWatch, S3, RDS, and third-party sources.

**Q70. What is a Transformer model and why does it matter for a Sales Architect?**
A: A Transformer is the neural network architecture underlying modern LLMs (GPT, Claude, Llama). The "attention mechanism" allows it to consider relationships between all words in a document simultaneously — unlike older sequential models. As a Sales Architect, you don't need to know the math, but understanding that: (1) transformers are stateless (each conversation requires full context), (2) they have context windows (limited text per request), (3) longer context = more tokens = more cost — helps you have credible conversations with technical buyers and explain why context window size matters for enterprise use cases.

# SECTION 3 — Multi-Cloud & Compliance (Q71–Q100)

**Q71. How do you compare AWS and Azure for enterprise security?**
A: AWS strengths: largest service catalog, most mature security services, most compliance certifications, best native integration between security tools. Azure strengths: best Microsoft ecosystem integration (Active Directory, M365, Windows), Sentinel is a best-in-class SIEM, strong hybrid cloud story (Azure Arc). Both are excellent — most enterprises use both. Positioning: if the customer is Microsoft-heavy (AD, M365), Azure Entra ID integration is strong; for everything else, AWS security breadth wins.

**Q72. What is Microsoft Sentinel and how does it compare to AWS native security?**
A: Sentinel is a cloud-native SIEM (Security Information and Event Management) + SOAR (Security Orchestration, Automation and Response). It collects logs from Azure, AWS, GCP, on-premises, and third-party tools, applies ML for threat detection, and enables automated playbooks for response. AWS doesn't have a direct native equivalent — Security Hub + GuardDuty + EventBridge + Lambda approximates SOAR capability, but Sentinel is more powerful as a dedicated SIEM. Many customers use Sentinel for SIEM while running workloads on AWS.

**Q73. What is Microsoft Entra ID (formerly Azure AD) and how does it integrate with AWS?**
A: Entra ID is Microsoft's cloud identity platform — SSO, MFA, Conditional Access for Microsoft and third-party apps. AWS integration: use Entra ID as a SAML 2.0 identity provider for AWS IAM Identity Center — employees sign in with their corporate Microsoft credentials and get temporary AWS access. This allows enterprises to maintain one identity store (Microsoft) while using both Azure and AWS, with centralized MFA and access policies.

**Q74. What is Google BeyondCorp and how does it relate to Zero Trust?**
A: BeyondCorp is Google's implementation of Zero Trust access — eliminates the traditional VPN model. Employees access internal apps based on device trust, user identity, and context — not network location. Google's enterprise product is BeyondCorp Enterprise (now part of Chrome Enterprise). AWS equivalent: AWS Verified Access. Mentioning BeyondCorp in customer conversations shows awareness of Zero Trust origins and helps position AWS Verified Access to customers familiar with Google's model.

**Q75. What is GDPR and which AWS services help with compliance?**
A: GDPR (General Data Protection Regulation): EU regulation protecting personal data. Key requirements: data minimization, purpose limitation, right to access, right to erasure, breach notification (72 hours), data residency. AWS services: Macie (find and classify PII), KMS (encrypt personal data), CloudTrail (audit who accessed what), Config (track data handling configuration), Artifact (GDPR data processing agreements), VPC (data residency — choose EU regions), Organizations (enforce EU-only regions via SCPs).

**Q76. What is HIPAA and what does AWS provide for HIPAA compliance?**
A: HIPAA (Health Insurance Portability and Accountability Act): US regulation protecting patient health information (PHI). Requirements: PHI encryption, access controls, audit logging, breach notification. AWS: signs Business Associate Agreements (BAA) with covered entities via Artifact, HIPAA-eligible services include EC2, S3, RDS, Lambda, KMS, CloudTrail, GuardDuty, and many more. Key: not all AWS services are HIPAA eligible — must use only eligible services for PHI.

**Q77. What is PCI DSS and which AWS services address its requirements?**
A: PCI DSS (Payment Card Industry Data Security Standard): 12 requirements for protecting cardholder data. Key requirements: network security (firewalls, no default passwords), cardholder data protection (encryption), access control (least privilege, MFA), monitoring (logs, IDS). AWS services: VPC + Security Groups (network segmentation), KMS (encryption), WAF + Shield (network protection), CloudTrail (audit logging), GuardDuty (intrusion detection), Config with PCI DSS conformance pack, Security Hub with PCI DSS standard.

**Q78. What is SOC 2 and how does AWS help customers achieve it?**
A: SOC 2 (Service Organization Control 2): auditing standard for service organizations covering 5 Trust Service Criteria: Security, Availability, Processing Integrity, Confidentiality, Privacy. AWS provides: SOC 2 reports (via Artifact) proving AWS infrastructure meets SOC 2 criteria, services customers use to build their own SOC 2 compliance (CloudTrail for logging, IAM for access control, KMS for encryption, Config for configuration management, Security Hub for continuous monitoring).

**Q79. What is FedRAMP and which customers need it?**
A: FedRAMP (Federal Risk and Authorization Management Program): US government program that standardizes security assessment and authorization for cloud services used by federal agencies. Customers who need it: US federal government agencies and their contractors. AWS GovCloud (US) regions are designed specifically for FedRAMP High workloads. AWS has numerous FedRAMP authorizations. Important for: defense contractors, civilian agencies, any company selling to the US government.

**Q80. What is ISO 27001 and how does it relate to AWS?**
A: ISO 27001: international standard for Information Security Management Systems (ISMS) — a systematic approach to managing sensitive company information. AWS is ISO 27001 certified across its global infrastructure and many services — customers can download the certification via AWS Artifact. Customers building on AWS can reference AWS's ISO 27001 certification in their own certification process (inheritance of controls from AWS's certified infrastructure).

**Q81. What is the NIST Cybersecurity Framework (CSF) and how does AWS map to it?**
A: NIST CSF: voluntary framework with 5 core functions: Identify (asset management), Protect (access control, encryption), Detect (monitoring, anomaly detection), Respond (incident response), Recover (recovery planning). AWS mapping: Identify → Config, Trusted Advisor; Protect → IAM, KMS, WAF, Shield; Detect → GuardDuty, CloudTrail, CloudWatch; Respond → Security Hub, EventBridge, Lambda; Recover → backups, DR architectures, AWS Elastic Disaster Recovery.

**Q82. What is a Business Associate Agreement (BAA) in the context of AWS?**
A: A BAA is a contract required under HIPAA between a covered entity (hospital, insurance company) and a business associate (vendor like AWS) that processes PHI. The BAA outlines how PHI must be handled, protected, and what happens in a breach. AWS BAAs are signed through AWS Artifact — the customer finds the BAA agreement in Artifact and accepts it online. Without a BAA, using AWS to process PHI is a HIPAA violation regardless of technical controls.

**Q83. What is the Cloud Security Alliance (CSA) and why is it relevant?**
A: CSA is a nonprofit organization promoting cloud security best practices. Key resources: Cloud Controls Matrix (CCM) — comprehensive framework of cloud security controls mapped to PCI DSS, HIPAA, ISO 27001, GDPR; STAR Registry — third-party assessed security of cloud providers. Relevant for sales: enterprise procurement often requires CSA STAR certification; mentioning CSA shows security credibility; AWS is CSA STAR certified.

**Q84. What is the CIS AWS Foundations Benchmark?**
A: A prescriptive set of security configuration recommendations for AWS accounts, published by the Center for Internet Security. Covers: IAM (MFA, password policy, access keys), logging (CloudTrail, Config, VPC Flow Logs), monitoring (CloudWatch alarms for critical events), networking (VPC, security groups, NACLs). AWS Security Hub includes CIS Benchmark as a built-in compliance standard — it automatically checks your account against all CIS controls and shows PASS/FAIL.

**Q85. What is the difference between a SIEM and a SOAR?**
A: SIEM (Security Information and Event Management): collects, aggregates, and analyzes security logs and events from across the environment — for detection and investigation. SOAR (Security Orchestration, Automation and Response): automates and orchestrates security response workflows — when a threat is detected, automatically run a playbook (isolate the instance, notify the team, create a ticket). AWS approximation: CloudWatch Logs + Security Hub = SIEM-like; EventBridge + Lambda + Systems Manager = SOAR-like. Microsoft Sentinel = SIEM + SOAR in one.

**Q86. What is Cloud Security Posture Management (CSPM)?**
A: CSPM continuously monitors cloud environments for misconfigurations and compliance violations. It scans resource configurations against security best practices and compliance frameworks, identifies risks (public S3 buckets, open security groups, missing encryption), and provides remediation guidance. AWS native CSPM: Security Hub + Config. Third-party CSPM tools: Wiz, Prisma Cloud by Palo Alto Networks, Lacework, Orca Security. The audit tool you built is a basic CSPM!

**Q87. What is Cloud Workload Protection Platform (CWPP)?**
A: CWPP protects cloud workloads (VMs, containers, serverless) at runtime — monitoring for threats, vulnerabilities, and anomalous behavior at the workload level. AWS native CWPP: Amazon GuardDuty (EC2, EKS, Lambda), Amazon Inspector (vulnerability scanning), Systems Manager (patch management). Third-party: CrowdStrike Falcon for AWS, Palo Alto Prisma Cloud. CSPM + CWPP together = comprehensive cloud security.

**Q88. What is the difference between data residency and data sovereignty?**
A: Data residency: where data is physically stored — the requirement to keep data within specific geographic boundaries (e.g., EU data must stay in EU). Data sovereignty: which country's laws apply to the data — data stored in a country is subject to that country's government requests. AWS addresses both: customers choose which AWS Region to store data in (data never leaves that region without customer action), and AWS has region-specific compliance certifications and government agreements.

**Q89. What is Google Chronicle and how does it compare to AWS logging?**
A: Chronicle is Google Cloud's cloud-native SIEM — stores security telemetry at scale with fast search, uses YARA-L detection rules for threat hunting, integrates with Google Threat Intelligence. Compare to AWS: AWS doesn't have a purpose-built SIEM — Security Lake provides the data lake, but customers need Athena for query or a third-party SIEM for full SIEM capabilities. Chronicle is often mentioned by customers evaluating GCP security alongside AWS.

**Q90. What is Wiz and how does it complement AWS Security Hub?**
A: Wiz is a cloud security platform (CSPM/CWPP) that agentlessly scans cloud environments by reading cloud provider APIs — no agents needed. It builds a security graph connecting all cloud resources and their relationships to identify attack paths. It integrates with AWS Security Hub (sends findings to Hub) and works across AWS, Azure, GCP, and Kubernetes. Complement: Wiz provides deeper cross-cloud visibility and attack path analysis; Security Hub aggregates findings centrally and runs compliance checks. Many enterprises use both.

**Q91. What is Azure Defender for Cloud and how does it compare to AWS Security Hub?**
A: Defender for Cloud (formerly Azure Security Center + Azure Defender): CSPM + CWPP for Azure and hybrid environments. Features: security score, recommendations, threat protection for VMs, containers, databases, and more. Compare to Security Hub: both provide a security posture dashboard with scores and recommendations. Defender for Cloud also protects non-Azure resources (AWS workloads via Arc), making it relevant for multi-cloud conversations. Security Hub is AWS-native with deeper AWS service integration.

**Q92. What is the difference between IDS and IPS?**
A: IDS (Intrusion Detection System): monitors network traffic and alerts on suspicious activity — detection only, does not block. IPS (Intrusion Prevention System): monitors and actively blocks suspicious traffic in real-time. AWS equivalents: GuardDuty = cloud-native IDS (detects and alerts, does not block directly); AWS Network Firewall = IPS capability (actively blocks traffic based on rules); WAF = application-layer IPS (blocks malicious HTTP requests).

**Q93. What is a landing zone in AWS?**
A: A landing zone is a pre-configured, secure multi-account AWS environment that follows best practices — a foundation for all workloads. Components: master account for billing, log archive account for centralized logging, security/audit account, organizational units (OUs) for different business units, baseline security controls (CloudTrail, Config, GuardDuty enabled everywhere, SCPs enforced). AWS Control Tower automates landing zone creation. Important for large enterprises starting or scaling their AWS journey.

**Q94. What is the difference between penetration testing and vulnerability scanning on AWS?**
A: Vulnerability scanning: automated tool checks for known vulnerabilities (CVEs) without exploiting them — Amazon Inspector does this. Penetration testing: human or automated tool actually attempts to exploit vulnerabilities to prove exploitability. AWS policy: customers can pen test their own AWS resources without prior approval for specific services (EC2, RDS, CloudFront, Aurora, API Gateway, Lambda, Lightsail, Elastic Beanstalk). Some activities (DDoS testing, port flooding) require advance permission from AWS.

**Q95. What is the AWS Customer Responsibility for security in managed services like RDS?**
A: For managed services (RDS, Lambda, ECS), AWS takes on more responsibility — AWS patches the database engine, manages OS, handles hardware failure. Customer still responsible for: IAM (who can access the database), network security (VPC placement, security groups), data encryption (enabling encryption at rest and in transit), database user credentials (rotate them), backup configuration, and application-level access controls. The shared responsibility model shifts but doesn't disappear for managed services.

**Q96. What is CASB (Cloud Access Security Broker)?**
A: CASB is a security policy enforcement point between cloud service users and cloud applications — it provides visibility into cloud app usage, enforces security policies, and protects data. Functions: discovery (what cloud apps are employees using?), compliance (ensure cloud usage meets policies), data security (DLP for cloud data), threat protection. AWS doesn't have a native CASB, but partners like McAfee MVISION Cloud, Microsoft Defender for Cloud Apps, and Netskope integrate with AWS.

**Q97. What is microsegmentation and how is it implemented in AWS?**
A: Microsegmentation divides the network into small, isolated zones with granular access controls — limits lateral movement if an attacker gains access. AWS implementation: VPC per application or environment, security groups to allow only required port/protocol combinations between specific resources (not broad subnet-level rules), private subnets for all non-public resources, VPC endpoints for service communication without internet, AWS Network Firewall for east-west traffic inspection within VPC.

**Q98. What is AWS PrivateLink?**
A: PrivateLink creates private, secure connectivity between VPCs, AWS services, and on-premises without exposing traffic to the public internet. Uses interface VPC endpoints (ENIs with private IPs). Use cases: privately access AWS services (S3, KMS, Bedrock) from within VPC without internet, expose SaaS services to customers privately, connect VPCs without VPC peering (avoids overlapping CIDR issues). Security benefit: data never traverses public internet, reducing exposure.

**Q99. What is AWS Network Firewall and when would you use it vs WAF vs Security Groups?**
A: Network Firewall: managed stateful/stateless firewall at VPC level — deep packet inspection, intrusion prevention, protocol-level filtering (block outbound DNS to untrusted servers), works on any IP traffic. WAF: HTTP/HTTPS application layer — filter web requests based on content, headers, IPs. Security Groups: basic instance-level firewall — allow/deny by IP and port. Use together: Security Groups for basic instance protection, WAF for web applications, Network Firewall for advanced network-level control (outbound filtering, lateral movement prevention).

**Q100. What is the MITRE ATT&CK framework and how do AWS security services map to it?**
A: MITRE ATT&CK is a comprehensive knowledge base of adversary tactics and techniques based on real-world observations — used to describe how attackers operate. AWS mapping: Initial Access (GuardDuty detects unusual console login, IAM credential theft), Execution (GuardDuty detects crypto-mining commands), Persistence (CloudTrail records new IAM user creation, backdoor access keys), Privilege Escalation (IAM Access Analyzer detects new admin policies), Exfiltration (GuardDuty detects unusual data transfer, Macie detects sensitive data access). Using ATT&CK vocabulary in customer conversations signals advanced security knowledge.

**Q125. What is a Security Operations Center (SOC) and how does AWS support it?**
A: A SOC is a centralized team and facility responsible for monitoring, detecting, investigating, and responding to security incidents. AWS supports SOC teams: Security Hub (centralized findings from all sources), GuardDuty (continuous threat detection feeding SOC), CloudTrail (audit logs for investigation), Amazon Detective (visualize and investigate findings), EventBridge + Lambda (automate tier-1 response, freeing SOC for complex investigations), Security Lake (data lake for threat hunting), CloudWatch (alerting and dashboards). Position: AWS security services can help a small team do the work of a large SOC.

**Q126. How do you handle a customer who says they'll "figure out security later"?**
A: "I hear that often, and I understand — when you're moving fast, security can feel like something you'll fix later. But the cost of adding security after the fact is 10x the cost of doing it right the first time. I've seen companies spend 6 months retrofitting encryption on databases that were built without it. More importantly, a breach before you have basic controls in place can be company-ending — not just financially, but reputationally." Then offer an easy starting point: "Let's start with just these three things that take 30 minutes and cost almost nothing: CloudTrail, GuardDuty, and MFA on all accounts. We can build from there."

**Q127. What are the 5 questions every Sales Architect should ask before a customer meeting?**
A: (1) What industry is the customer in and what compliance requirements do they have? (2) What is the stated business problem or opportunity? (3) Who will be in the meeting — technical, business, or both? (4) What is the customer's current AWS usage and security maturity? (5) What does a win look like for this meeting — discovery, demo, POC agreement, or proposal?

**Q128. How would you position AWS Bedrock to a customer concerned about AI vendor lock-in?**

If Anthropic's Claude serves you best today but Meta's Llama improves and becomes better for your use case in six months, you switch models without changing your application code. **Q129. What is the difference between a Sales Engineer, Solutions Architect, and a Technical Account Manager?**
A: Sales Engineer (SE) / Pre-Sales Solutions Architect: works with Account Executives before a sale — technical qualification, demos, proposals, POCs, competitive differentiation. Goal: win the deal. Technical Account Manager (TAM): post-sales — works with existing customers to maximize their AWS usage, proactively identifies opportunities and issues, advocacy and escalation. Goal: retain and grow existing accounts. Solutions Architect (post-sales): helps customers architect and build solutions after they've purchased. Some companies blur these roles, but the pre/post-sales distinction is key.

**Q130. How would you prepare for a security briefing with a Fortune 500 CISO?**
A: Research: their industry, public security incidents they've had, compliance requirements for their sector, company size and AWS footprint (if known). Prepare: 3 business outcomes they care about (reducing breach risk, achieving compliance, reducing tool sprawl), relevant customer case studies in their industry, security posture assessment framework you'll use in the meeting. Meeting structure: 5 min — relationship building; 10 min — understanding their priorities (ask, don't tell); 15 min — share relevant insights and case studies; 15 min — demo or whiteboard most relevant solution; 15 min — next steps. Never sell — advise.

**Q131. What is a cloud security maturity model?**
A: A framework assessing an organization's cloud security sophistication: Level 1 (Basic): ad-hoc security, minimal controls, no logging; Level 2 (Reactive): basic controls (MFA, some logging), responds to incidents but no proactive detection; Level 3 (Proactive): GuardDuty, CloudTrail, Config enabled, compliance checks running, incident response plan exists; Level 4 (Optimized): Security Hub across all accounts, automated remediation, threat hunting, continuous compliance, security integrated into DevOps (DevSecOps); Level 5 (Advanced): Zero Trust implementation, AI-powered threat detection, security as code, continuous improvement program. Use this framework to assess customers and position next steps.

**Q132. How would you explain the value of AWS Security Hub to a 10-year-old?**
A: "Imagine you have 10 different security cameras around your house, but each camera has its own tiny screen in a different room. You'd have to run to 10 different rooms to check if everything is safe. Security Hub is like one giant TV screen in the living room that shows all 10 cameras at once — and if anything looks suspicious on any camera, a big red alarm goes off right there. You can see everything from one place without running around."

**Q133. What is the "shared fate" model that some cloud providers discuss?**
A: Shared fate (proposed by Google) goes beyond shared responsibility — the cloud provider has a stake in the customer's security outcomes, not just their own infrastructure. In practice: AWS provides security best practice guidance (Well-Architected Framework), default-on security features (default encryption, S3 Block Public Access), compliance programs, professional services, and AWS Shield DDoS financial protection as forms of shared fate. Position: AWS invests heavily in making customers secure because AWS's reputation depends on it.

**Q134. How do you position your AWS Security Audit Tool on GitHub in a job interview?**
A: "I built an AWS Security Audit Tool in Python using the boto3 SDK that performs 10 automated security checks across an AWS account — checking for common misconfigurations like open security groups, missing CloudTrail, no MFA on IAM users, public S3 buckets, and more. It runs from the command line, produces a scored report, and is hosted on GitHub. When I ran it against my own account, it found 7 genuine security issues including 17 security groups exposing SSH to the internet. This project demonstrates my ability to turn security knowledge into working tools — exactly what customers need from a Sales Architect who can speak credibly to engineering teams."

**Q135. What would you do if a customer asks a technical question you don't know the answer to?**
A: "Great question — I want to make sure I give you an accurate answer rather than guessing. Let me confirm the details and get back to you by [specific time]." Then follow up promptly with the correct answer, including the source (AWS documentation link). This is far better than providing incorrect information — trust is the foundation of the relationship. Also: bring a technical specialist for complex architecture conversations — knowing when to get help is itself a valuable skill.

**Q136. How does DevSecOps relate to cloud security and how would you sell it?**
A: DevSecOps integrates security into every stage of the software development lifecycle instead of checking security at the end. AWS tools for DevSecOps: CodeGuru Security (scans code for vulnerabilities as developers write it), Inspector (scans container images in ECR before deployment), CodePipeline + security checks (block deployment if vulnerabilities found), IAM permission boundaries (developers can't exceed security limits), Config Rules (catch misconfigurations immediately at deployment). Sales angle: "Shift security left — finding a vulnerability in development costs $80 to fix; in production it costs $8,000."

**Q137. What is threat modeling and how would a customer use AWS tools to support it?**
A: Threat modeling: systematically identifying potential security threats to a system and designing controls to mitigate them. Process: identify assets, identify threats (using frameworks like STRIDE or MITRE ATT&CK), identify controls, prioritize by risk. AWS tools: AWS Threat Composer (diagramming tool for threat models), Well-Architected Tool (structured risk assessment), Inspector (identifies actual vulnerabilities matching theoretical threats), GuardDuty (detects real-world attacks matching threat models). Sales angle: "Let's run a threat modeling exercise for your planned architecture before you build it."

**Q138. What is the difference between authentication and authorization?**
A: Authentication: verifying identity — "who are you?" (username + password, MFA, certificate). Authorization: verifying permissions — "what are you allowed to do?" (IAM policies, S3 bucket policies, resource policies). In AWS: authentication happens when you log into the console or call an API; authorization happens when AWS evaluates IAM policies to decide if your request is allowed. Common mistake: strong authentication (MFA) doesn't help if authorization is too permissive (AdministratorAccess for everyone).

**Q139. How do you explain the concept of "security as code"?**
A: Security as code means defining security controls in code rather than configuring them manually — version-controlled, testable, repeatable. Examples: CloudFormation/Terraform templates that always deploy EC2 with encryption enabled and specific security group rules; Config rules that automatically enforce security standards; SCPs written as code and deployed via CI/CD pipeline; IAM policies version-controlled in Git. Benefits: consistent security across environments, security changes go through code review, auditable history of security decisions, automatic application of controls to new resources.

**Q140. What is lateral movement and how do AWS security services detect it?**
A: Lateral movement: after compromising one system, an attacker moves to other systems within the environment to expand access or reach higher-value targets. AWS detection: GuardDuty detects unusual API calls from EC2 instances (an EC2 suddenly making IAM calls — unusual), VPC Flow Logs + GuardDuty detect unusual internal network communication patterns, CloudTrail detects credentials being used from unexpected sources, Security Hub correlates findings to identify attack chains. Defense: VPC micro-segmentation, IAM least privilege (even if one instance is compromised, it can't access everything).

**Q141. What is the "assume breach" mindset and how does AWS support it?**
A: Assume breach: design systems assuming attackers will eventually get in — focus on detection, containment, and response rather than just prevention. AWS support: GuardDuty for detection once inside, VPC segmentation to limit blast radius, IAM least privilege to limit what a compromised identity can access, CloudTrail to log all actions for forensics, automated response via EventBridge + Lambda to contain threats immediately, AWS Backup for recovery. Sales angle: "Prevention fails eventually — what happens when it does? That's where GuardDuty, CloudTrail, and automated response make the difference."

**Q142. How would you approach selling to a customer who is skeptical of AI security tools?**
A: Acknowledge their skepticism: "You're right to be thoughtful — not all AI security tools are equal." Then educate: GuardDuty's ML establishes behavioral baselines specific to each customer's environment — it doesn't just pattern-match like rules do. Share proof: GuardDuty found threats that rule-based systems missed in [specific example]. Address the false positive concern: GuardDuty findings have severity levels and explanations, findings integrate with Security Hub where teams can tune and suppress. Offer: "Let's run GuardDuty for 30 days — first month is on us through the free trial — and review what it finds. Judge it on results."

**Q143. What is an AWS Partner and how does it relate to a Sales Architect career?**
A: AWS Partners are companies in the AWS Partner Network (APN) that build on, sell, or consult on AWS. Types: Technology Partners (ISVs building software on AWS), Consulting Partners (SIs, managed service providers, advisory firms). Sales Architects often work at AWS Partners: helping customers adopt AWS, building AWS-based solutions, and earning AWS partner certifications. Tier levels: Registered → Select → Advanced → Premier based on certifications, customer success, and revenue. Being at an AWS Premier Consulting Partner carries significant market credibility.

**Q144. How do you handle price objections for AWS security services?**
A: (1) Reframe cost as investment: "The question isn't whether GuardDuty is worth $500/month — it's whether you can afford not to detect a $5M breach for 6 months." (2) Show savings vs alternatives: GuardDuty + Security Hub vs hiring a dedicated security analyst ($120K/year). (3) Start small: "Let's enable GuardDuty in one account for 30 days — free trial — and see what it finds. Then we'll size the real cost based on your actual data volume." (4) Reference breaches in their industry to make risk concrete. (5) Calculate specific ROI for their business using their revenue and data sensitivity.

**Q145. What is AWS Marketplace and how does it benefit security customers?**

**Q146. How would you build a 90-day plan as a new Tech Sales Architect joining a company?**

**Q147. How do you quantify the risk of having no CloudTrail enabled?**
A: Without CloudTrail: (1) You cannot answer "who deleted that database?" — no audit trail means no forensics capability; (2) You fail nearly every compliance audit — PCI DSS, HIPAA, SOC 2 all require audit logging; (3) Mean time to detect and respond to incidents increases dramatically — security teams are investigating blind; (4) You cannot prove compliance to regulators — potential fines; (5) Cyber insurance claims may be denied without audit logs proving you had adequate controls. Cost of CloudTrail: first trail is free. Cost of no CloudTrail: failed audit, compliance fine, unresolvable security incident.

**Q148. What are the top 5 security mistakes companies make when moving to AWS?**
A: (1) Using the root account for daily operations and not enabling MFA — most critical; (2) Overly permissive IAM policies (AdministratorAccess for developers) — violates least privilege; (3) Not enabling CloudTrail — no audit trail; (4) Public S3 buckets containing sensitive data — major breach cause; (5) Security groups open to 0.0.0.0/0 on SSH/RDP — massive attack surface. These are exactly what your audit tool checks for — making it a great conversation starter: "Our tool found these issues in our own account on the first run."

**Q149. How do you position AWS security for a company that already has significant on-premises security investment?**
A: "Your on-premises security investment is valuable — we help you extend it to the cloud, not replace it. Your existing SIEM? Security Hub and CloudTrail logs feed directly into Splunk and other SIEMs. Your Active Directory? IAM Identity Center federates with it — same credentials, same access control model. Your firewall vendor? Many integrate with AWS via the Marketplace. And for workloads that are uniquely cloud — like the API calls your developers make to create and modify infrastructure — we add capabilities your on-premises tools can't cover, like GuardDuty's analysis of CloudTrail."
